# Methodology

## Project Overview

This project implemented an encoder-based NeFT workflow for multilingual
natural language processing experiments. The repository was organized as
a staged pipeline comprising experiment configuration, baseline
fine-tuning, NeFT mask generation, Probeless mask preparation, masked
fine-tuning, evaluation, and post-processing of evaluation summaries
into plots. The workflow was defined around experiment keys in
`config.py`, and each experiment key determined the backbone model,
dataset source, and ranking source used in subsequent stages.

The end-to-end pipeline followed a consistent order. First, a fully
trainable baseline model was fine-tuned for the target task. Second,
NeFT masks were generated by comparing the pretrained encoder with the
baseline checkpoint, while Probeless masks were produced separately from
precomputed ranking files. Third, masked fine-tuning was performed for a
fixed sweep of neuron percentages. Finally, each trained checkpoint was
evaluated on the held-out test split, summary CSV files were generated,
and metric plots were produced from those summaries.

## Experimental Scope

The experimental scope covered three task families: part-of-speech
tagging, named entity recognition, and sentiment classification. Two
multilingual encoder backbones were used throughout the repository:
mBERT (`bert-base-multilingual-cased`) and XLM-R (`xlm-roberta-base`).
Across these tasks and models, three methods were compared: the baseline
full fine-tuning approach, NeFT-based masked fine-tuning, and
Probeless-based masked fine-tuning. The masked methods were executed for
a fixed percentage sweep of 5, 10, 15, 20, 25, 30, 35, 40, 45, and 50
percent, while the baseline served as the unconstrained reference
configuration.

The experiment has a multilingual scope including Hindi, Arabic,
Chinese, and French across the task families supported by the
repository. For each experiment key, the project specifies the expected
baseline checkpoint directory, NeFT mask path pattern, Probeless mask
path pattern, masked-training output directory patterns, evaluation
summary CSV path, and plot directory. This structure provided a uniform
naming convention for all runs, regardless of task or backbone model.

## Data and Task Setup

The repository used two distinct data-handling pipelines, depending on
whether the task was token classification or sequence classification.
POS and NER experiments were handled through a shared
token-classification workflow, whereas sentiment experiments used a
separate sequence-classification workflow. This distinction was
reflected both in the input file formats and in the training and
evaluation scripts selected for each task family.

For POS and NER, the input data were read from CoNLL-U or CoNLL-U-like
files with dedicated train, development, and test splits. The scripts
processed sentence boundaries directly from the file structure, skipped
comment lines, and ignored multiword or decimal token identifiers.
Within each valid token row, the token text was read from column 2 and
the label was read from column 4. Label mappings were constructed from
the training set, and tokenization was performed with
`is_split_into_words=True` so that labels could be aligned to
subword-tokenized inputs. Only the first subtoken of each original token
received the label, while later subtokens were assigned ignore indices.
This same data-reading and label-alignment approach was used for both
POS and NER, with the task difference determined by the configured
dataset path and experiment key rather than by separate modeling logic.

For sentiment classification, the input data were read from CSV files
containing `text` and `label` fields. The shared split utility created
deterministic train, validation, and test partitions using a 70/15/15
split with seed 42. Rows with missing text or label values were removed,
labels were cast to integer form, and label mappings were derived from
the training split. The text inputs were tokenized directly as raw
strings rather than as word lists, and the test split remained held out
during training and masked fine-tuning.

At the configuration level, `config.py` served as the central registry
for all experiments. It defined the root directories for datasets,
rankings, and masks; the model-name constants for mBERT and XLM-R; and
the per-experiment metadata needed by downstream scripts. For token
tasks, the configuration stored dataset-folder and file-prefix
information, whereas sentiment experiments stored the CSV filename. It
also linked each experiment to a ranking file used for Probeless mask
preparation.

## Baseline Fine-Tuning

Baseline fine-tuning was implemented as the fully trainable reference
stage for each experiment. For POS and NER, baseline training was
performed by `baseline/train_baseline.py`. The script accepted an
experiment key through the `–lang` argument, loaded the corresponding
dataset metadata from `config.py`, read the train and development
CoNLL-U files, built label mappings from the training data, tokenized
with a maximum sequence length of 256, and fine-tuned an
`AutoModelForTokenClassification` model initialized from the configured
backbone. Validation metrics were computed with `seqeval`, and model
selection was based on validation F1. Baseline checkpoints were saved
under `./output/{lang}-baseline`.

For sentiment experiments, baseline training was performed by
`baseline/train_baseline_sentiment.py`. This script used the configured
sentiment CSV, generated deterministic train, validation, and test
splits, and fine-tuned an `AutoModelForSequenceClassification` model
from the configured backbone. Tokenization again used a maximum length
of 256, but the validation metrics in this branch were accuracy and
weighted F1 rather than the `seqeval` metrics used for token
classification. As in the token-classification baseline, the checkpoint
was written to `./output/{lang}-baseline`.

The baseline stage used fixed training settings within each task family.
Token-classification baseline training used seed 42, 20 training epochs,
train and evaluation batch sizes of 8 and 16, a learning rate of `2e-5`,
epoch-level evaluation and checkpoint saving,
`load_best_model_at_end=True`, early stopping with patience 5, and a
save limit of 2 checkpoints. Sentiment baseline training used the same
number of epochs and learning rate, but used train and evaluation batch
sizes of 16 and 32 and a logging interval adapted for the sentiment
workflow. These baselines established the reference checkpoints used
later in mask generation and comparison.

## NeFT Neuron Selection / Mask Generation

NeFT mask generation was performed by `find_neuron/find_neuron.py`. This
stage took the configured pretrained backbone model and the
corresponding baseline checkpoint as input and compared their encoder
parameters in order to derive binary masks for masked fine-tuning. The
implementation operated on the encoder feed-forward output projection
weights, specifically `output.dense.weight` for each encoder layer. The
script loaded both models through `AutoModel`, iterated over
`model.encoder.layer`, and computed row-wise cosine similarity between
the pretrained and baseline versions of `output.dense.weight`. These
cosine values were written to `cos_out.csv` under the NeFT mask
directory.

The percentage parameter was converted into a target count using the
fixed expression `int(12*768*p/100)`, reflecting the hardcoded
assumptions of 12 layers and 768 neurons per layer. The script sorted
the cosine scores, determined a threshold for the requested percentage,
and selected entries with cosine score below that threshold. The
resulting selection was encoded as binary tensors keyed by layer index
in the form `{layer_idx}_out`. One NeFT mask file was produced for each
percentage in the sweep from 5 to 50, and all such files were written
under `./masks/{lang}/baseline_neft/` using the filename pattern
`{p}neft_mask.pt`. In this repository, the masks therefore represented
selected positions associated with encoder FFN `output.dense.weight`
rather than attention weights, activation statistics, or classifier-head
parameters.

## Probeless Mask Preparation

Probeless mask preparation was implemented in
`prepare_probeless/prepare_masks.py`. This script converted an existing
ranking file for a given experiment into binary mask tensors suitable
for the masked-training stage. The ranking file path was resolved
through `config.py`, and the script used the configured mask directory
`./masks/{lang}/` as its output location. Unlike the NeFT stage, this
script did not compare pretrained and fine-tuned models. Instead, it
took ranking file from probeless ranking methodology and used them to
create a neuron mask.

The ranking data were loaded from a pickle file and transformed into a
tabular structure from which layer indices and neuron indices were
derived using integer division and modulus with respect to 768. The
script removed entries assigned to layer 0, shifted the remaining layer
numbering down by one, and selected the first `int(768*12*p/100)` ranked
entries for each percentage, using the fixed total-neuron assumption of
9,216. For each percentage, it grouped neurons by layer and created
flattened binary masks keyed as `{layer}_out`, with tensor length
corresponding to `768*3072`. The resulting files were saved as
`5probeless_mask.pt` through `50probeless_mask.pt` under
`./masks/{lang}/`. In this repository, Probeless masks thus represented
binary selections over encoder FFN output-projection positions, aligned
with the same masked-parameter family used in NeFT training.

## Masked Fine-Tuning

Masked fine-tuning constituted the main comparison stage for NeFT and
Probeless. For both methods, separate training scripts existed for
token-classification tasks and sentiment classification, but the masking
mechanism remained structurally consistent across these branches. Each
masked run was executed independently for one of the predefined
percentages 5, 10, 15, 20, 25, 30, 35, 40, 45, or 50, and each run
produced its own checkpoint directory. NeFT outputs followed the pattern
`./output/{lang}-neft/{p}percent/`, whereas Probeless outputs followed
`./output/{lang}-probeless/{p}percent/`.

For POS and NER, NeFT masked training was implemented in
`train_neuron/train_neft.py`, and Probeless masked training was
implemented in `train_neuron/train_probeless.py`. Both scripts reused
the same CoNLL-U data parser and subtoken-alignment logic as baseline
token training. For each percentage, the script loaded the corresponding
mask file, reinitialized an `AutoModelForTokenClassification` model from
the pretrained backbone specified in the experiment configuration, and
trained it on the train split while validating on the development split.
The masking mechanism was applied through a callback executed at
`on_pre_optimizer_step`, where the gradients of
`layer.{i}.output.dense.weight` were multiplied elementwise by a
reshaped mask tensor of size `(768, 3072)`. This implementation meant
that masked training in the repository did not begin from the baseline
checkpoint; instead, each percentage run started from the pretrained
backbone and constrained gradient updates only for the selected FFN
output-projection parameters. Other trainable parameters remained
active.

For sentiment classification, NeFT masked training was implemented in
`train_neuron/train_neft_sentiment.py`, and Probeless masked training
was implemented in `train_neuron/train_probeless_sentiment.py`. These
scripts used the same deterministic train/validation/test split protocol
as the sentiment baseline stage. For each percentage, the model was
reinitialized from the configured pretrained backbone, tokenized inputs
were prepared with maximum length 256, and masked fine-tuning was
performed using the same gradient-masking callback applied to encoder
`output.dense.weight`. The difference between NeFT and Probeless in this
branch was therefore the source of the binary mask, not a difference in
the training framework itself. Validation during training was based on
accuracy and weighted F1, consistent with the sentiment baseline stage.

The fixed training settings for masked fine-tuning matched the baseline
settings within each task family, with minor differences in
checkpoint-retention limits between NeFT and Probeless scripts.
Token-classification masked runs used 20 epochs, a learning rate of
`2e-5`, train and evaluation batch sizes of 8 and 16, epoch-level
evaluation and saving, early stopping with patience 5, and validation F1
as the model-selection metric. Sentiment masked runs used the same
number of epochs and learning rate but used batch sizes of 16 and 32 and
selected checkpoints using validation F1 under the sentiment-training
branch. The uniform percentage sweep and path conventions enabled direct
downstream evaluation of all masked variants.

## Evaluation Protocol

Final evaluation was conducted through separate scripts for
token-classification and sentiment tasks, and the outputs of both
scripts were standardized into CSV summary files stored under
`./Results/scores/`. These summary files served as the direct source for
report tables and plots. In all cases, the baseline was recorded with
percentage 0, and masked variants were evaluated for the percentage
values from 5 to 50.

For POS and NER, evaluation was implemented in
`evaluation/evaluate_model.py`. The script read the held-out test split
from the configured CoNLL-U file, parsed valid token rows, read tokens
from field 2 and labels from field 4, tokenized the test data with
maximum length 256, aligned labels to first subtokens, and evaluated the
baseline, NeFT, and Probeless checkpoints from their expected output
directories. Final metrics were computed with `sklearn` over non-ignored
positions and written to
`./Results/scores/evaluation_summary_{lang}.csv`. The reported columns
were `Model_Type`, `Percent`, `Accuracy`, `F1_Macro`, `Precision`, and
`Recall`, corresponding to token-level accuracy, macro precision, macro
recall, and macro F1.

For sentiment classification, evaluation was implemented in
`evaluation/evaluate_model_sentiment.py`. This script reloaded the
held-out test split through the same split utility used during training,
tokenized the texts once using the configured tokenizer, and evaluated
baseline, NeFT, and Probeless checkpoints from the expected output
directories. The evaluation summary file was again written to
`./Results/scores/evaluation_summary_{lang}.csv`, but the sentiment
summary included `F1_Weighted` in addition to `Accuracy`, `F1_Macro`,
`Precision`, and `Recall`. Accordingly, the reported columns were
`Model_Type`, `Percent`, `Accuracy`, `F1_Macro`, `F1_Weighted`,
`Precision`, and `Recall`.

Plot generation was handled by `evaluation/plot.py`. This script read
the evaluation summary CSV for a given experiment key, normalized the
model-type labels to Baseline, NeFT, and Probeless, and produced
comparison plots saved under `./Results/metric_plots/{lang}/`. In the
plotting routine, the baseline was rendered as a horizontal reference
line, while the masked methods were plotted as performance curves over
the percentage sweep. By default, the plotting configuration enabled
Accuracy, Precision, Recall, and F1_Macro, while F1_Weighted remained
disabled unless manually activated. These plots were therefore
post-processing artifacts derived entirely from the evaluation summary
CSV files.
